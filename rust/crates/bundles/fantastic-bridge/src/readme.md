# ws_bridge / cloud_bridge ‚Äî cross-kernel comms (two io_bridge derivations)
TWO bundles share one engine (mirrors py's separate `ws_bridge` + `cloud_bridge`): `ws_bridge.tools` opens a WS to the remote kernel's `web_ws` surface (transports: memory (tests) / ws / ssh+ws); `cloud_bridge.tools` dials OUT to a zero-trust relay (both peers pair + forward opaque frames; peer‚Üîpeer TLS 1.3 mutual auth with self-signed Ed25519 device certs pinned by PUBLIC KEY, so the relay sees only ciphertext). Each derivation only opens transports in its own family. A bridge ships raw `{type:"call", id, target, payload}` frames; the remote dispatches `kernel.send` exactly like any peer/WS client and replies over the same socket ‚Äî no peer bridge needed. Streaming via `watch_remote` (`{type:"watch", src}` out, `{type:"event"}` back, re-emitted on the bridge's own inbox). Weak binding ‚Äî the remote is addressed by URL + path only, no shared types. Weak proxy ‚Äî local‚Üílocal stays direct `kernel.send`. **Authorization** ‚Äî two symmetric, typed per-leg rules (enforced on the receiver): `ingress_rule` (inbound FILTER at one choke point) and `egress_rule` (outbound DECORATOR, stamps a credential on the frame envelope); a legacy `auth` shorthand sets both. Types: `allow_all` (default ingress) / `silent` (default egress) | `deny_inbound` (refuse inbound `call`s, reply `{reason:"unauthorized"}`) | `password` (kernel-GROUP shared secret ‚Äî ingress checks the envelope `auth_token` against the group token from an env var, default `FANTASTIC_GROUP_TOKEN`; egress presents it). Resolved by name from the `authorizer::ingress` / `authorizer::egress` registries (one module per rule).

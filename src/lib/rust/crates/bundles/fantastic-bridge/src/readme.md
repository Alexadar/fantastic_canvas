# ws_bridge / relay_connector — cross-kernel comms (two io_bridge derivations)
TWO bundles share one engine (mirrors py's separate `ws_bridge` + `relay_connector`): `ws_bridge.tools` opens a WS to the remote kernel's `web_ws` surface (transports: memory (tests) / ws / ssh+ws); `relay_connector.tools` dials a relay-KERNEL router (`../fantastic_relay`) at `ws://<host>/<guid>` (group password in `X-Fantastic-Auth`, subprotocol `fantastic.relay.v1`) and reaches a `partner_guid` — the relay routes by `target` and delivers peer→peer as one-way `{type:"event", source, payload}`. The connector TUNNELS bridge frames over that (`{type:"send", target:partner, payload:<frame>}` out, `event` in) so forward/reply correlation + symmetric serving work UNCHANGED; raw `read_stream` chunks ride as native binary WS frames (`[len|header|body]`, no base64). Each derivation only opens transports in its own family. A bridge ships raw `{type:"call", id, target, payload}` frames; the remote dispatches `kernel.send` exactly like any peer and replies over the same socket. Streaming via `watch_remote` (`{type:"watch", src}` out, `{type:"event"}` back, re-emitted on the bridge's own inbox). Weak binding — the remote is addressed by URL/GUID only, no shared types. Weak proxy — local→local stays direct `kernel.send`. **Authorization** — two symmetric, typed per-leg rules (enforced on the receiver): `ingress_rule` (inbound FILTER at one choke point) and `egress_rule` (outbound DECORATOR, stamps a credential on the frame envelope); a legacy `auth` shorthand sets both. Types: `deny_inbound` (**the DEFAULT ingress — SEALED by default: an absent `ingress_rule` refuses every inbound `call`/`watch`/`emit`, reply `{reason:"unauthorized"}`; open with `ingress_rule=allow_all`**) / `allow_all` / `silent` (the default egress — presents nothing) | `password` (kernel-GROUP shared secret — ingress checks the envelope `auth_token` against the group token from an env var, default `FANTASTIC_GROUP_TOKEN`; egress presents it). These per-leg rules gate the tunneled bridge calls and are INDEPENDENT of the relay's own connection auth. Resolved by name from the `authorizer::ingress` / `authorizer::egress` registries (one module per rule). The relay leg is self-healing: it auto-connects (eager + background) and auto-reconnects with the `reconnect` backoff (seconds, default 10; `0` = one-shot), sends a no-reply `keepalive` every `heartbeat` seconds, and `reflect.connected` tracks the live socket. It also exposes the relay DIRECTORY (the relay's own `relay` agent, `target:"relay"`, not the partner): `list_peers` → `{peers:[{guid,status,last_seen,since,attrs}]}`, and `watch_directory`/`unwatch_directory` re-emit `peer_joined`/`peer_left`/`peer_evicted`/`peer_status`/`peer_updated` events on the connector's own inbox. **Directory typing (kernelgroup):** a connector advertises an opaque **attrs blob** to the relay on connect (re-announced on each reconnect) — well-known keys `role` (`manager`|`kernel`, default kernel), `owner_guid` (managing peer, null=standalone), `exposes` (control-surface list); the relay stores + reflects it (into `list_peers` + `peer_updated`) and never interprets it. `set_identity` (verb: optional role/owner_guid/exposes, merged) updates them live + persists. reach ≠ control — driving a kernel owned by another manager is a plain `forward` to that manager. relay_connector record fields: `relay_url` · `guid` (auto-minted + persisted on first boot if absent — explicit wins, never regenerated; `partner_guid` stays explicit, discovered via the partner's reflect/`list_peers`) · `partner_guid` · `relay_token` (X-Fantastic-Auth) · `heartbeat` (s, default 30) · `reconnect` (s, default 10) · `role` · `owner_guid` · `exposes`.
